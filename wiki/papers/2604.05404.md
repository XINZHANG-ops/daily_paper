# Beyond Accuracy: Unveiling Inefficiency Patterns in Tool-Integrated Reasoning

**Published**: 2026-04-06 | **Added**: 2026-04-08 | **arXiv**: [2604.05404](http://arxiv.org/pdf/2604.05404)

## Summary

This paper identifies a critical blind spot in Tool-Integrated Reasoning (TIR) evaluation: while accuracy benchmarks are well-established, efficiency metrics that reflect real-world inference latency are largely absent. Existing metrics like token counts and tool-call counts fail to capture the asymmetric costs of compute-bound prefill and memory-bound decode phases, particularly the KV-Cache eviction during tool calls and the memory overhead from long tool responses that inflate context length.

The authors introduce PTE (Prefill Token Equivalents), a hardware-aware metric that unifies internal reasoning and external tool-use costs by modeling the prefill-decode asymmetry inherent in transformer inference. PTE achieves a correlation of r=0.9253 with wall-clock latency compared to r=-0.3750 for naive token counts, and maintains consistent model efficiency rankings across diverse hardware profiles (Spearman ρ>0.95). Through extensive experiments across five TIR benchmarks, the paper identifies four inefficiency patterns (Confirmatory Tool Usage, Tool-Mixing, Lack of Tool Priors, Tool Format Collapse) with cost multipliers ranging from 1.77x to 2.42x. A striking finding is that trajectories with higher PTE costs tend to have lower reasoning correctness, indicating that simply using more tools does not improve answer quality.

## Key Contributions

- **PTE Metric**: Hardware-aware unified metric for prefill and decode costs, directly modeling the memory-bound nature of decode phase in TIR
- **Strong Latency Correlation**: r=0.9253 with wall-clock latency vs r=-0.3750 for token counts; validated on 8xH200 cluster with 256 parallel requests
- **Hardware Robustness**: Spearman rank correlation >0.95 across H100, H200, A100, RTX 4090, V100
- **Four Inefficiency Patterns**: Identified and quantified with cost multipliers 1.77x-2.42x
- **Quality Trade-off Discovery**: Higher PTE costs correlate with lower reasoning correctness

## Method

### The Physical Reality of LLM Inference Cost

LLM inference exhibits phase-dependent bottlenecks:

**Prefill Phase (Compute-Bound)**: Input tokens are processed in parallel; cost dominated by matrix multiplications, proportional to model size Nparams. Estimated as Cprefill ≈ 2·Nparams [FLOPs].

**Decode Phase (Memory-Bound)**: Output tokens generated sequentially; performance limited by HBM bandwidth to load model weights and KV-Cache. Cost grows linearly with cumulative context length Lseq.

**In TIR Context**:
- Tool calls create pauses that evict KV-Cache, forcing recomputation
- Long tool responses inflate context, making each subsequent decode step more expensive

### PTE Definition

For a reasoning trajectory with k turns:
```
PTE = Σ (Dprefill_i + γ · Lseq_i · Ddecode_i)
```
Where:
- Dprefill_i: Total context tokens up to turn i (compute-bound cost)
- Ddecode_i: Tokens generated in turn i
- Lseq_i: Cumulative sequence length before decode in turn i
- γ: Ratio of decode to prefill cost (hardware-dependent coefficient)

### The γ Coefficient

γ represents the relative penalty of memory-bound to compute-bound operations:
```
γ = (2·nlayers·dmodel·HOI) / Nparams
```
Where HOI is Hardware Operational Intensity (FLOPs/Byte). For Grouped Query Attention (GQA), γ is scaled by KV-to-Query head ratio. For Multi-Head Latent Attention (MLA), dmodel is substituted with compressed dimensions.

### Validation Experiments

High-concurrency evaluation on 8xH200 GPU cluster with DeepSeek-V3.2:
- 256 parallel requests simulating industrial TIR workload
- Pure model generation latency (excluding tool execution, network transmission)
- Pearson correlation: PTE r=0.9253 vs Token Count r=-0.3750

## Results

### Inefficiency Patterns

**1. Confirmatory Tool Usage** (81% frequency in Qwen3-235B-Instruct on MATH500, 1.77x multiplier)
- Model solves problem internally, arrives at correct answer, then calls Python tool "to verify" known result
- Creates "first-step effect": heavy initial reasoning delays tool use, accumulating context
- Every subsequent step becomes more expensive due to non-reusable KV-Cache

**2. Tool-Mixing** (59% frequency in DeepSeek-V3.1-Term on WebInstruct, 2.42x multiplier)
- Trajectories use multiple distinct tool types (search + Python for same task)
- Accumulates intermediate output context without accuracy benefit
- Example: alternating between search and Python when single toolset suffices

**3. Lack of Tool Priors** (33% frequency in Qwen2.5-72B-Instruct on AIME24, 2.15x multiplier)
- Tool call returns empty output due to missing print statements or syntax errors
- Suggests limited pretraining exposure to Python tool environment
- Wasted inference steps without useful computation

**4. Tool Format Collapse** (100% frequency in Tongyi-Deepresearch on SimpleQA)
- Schema hallucination: model uses tool format from training that differs from system prompt
- Example: sends query as string instead of expected array; adds extra linefeeds breaking parser
- Causes immediate parsing failures, cascading errors

### Cost Analysis

**Per-Step PTE Escalation**: Unlike token-based "front-loading" patterns where costs appear early, PTE reveals that actual hardware costs escalate in later steps due to context accumulation. Token generation front-loads budgets; PTE exposes the true cost trajectory.

**Correct vs Incorrect Trajectories**: Incorrect trajectories consistently exhibit higher PTE across models and benchmarks. For example, on MATH500 Level 1:
- GPT-OSS: +5.1% PTE increase for incorrect
- GLM-4.5-Air: +419.9% increase
- Qwen3-32B: +793.2% increase

**Vast Cost Gap**: Models with similar accuracy (e.g., ~70% on AIME24) show PTE values spanning an order of magnitude or more.

### Quality vs Efficiency Trade-off

Thinking mode analysis on Qwen3-235B-Thinking:
- On AIME25: +16.7% accuracy gain at only 1.8x PTE cost (beneficial)
- On SimpleQA: -3.4% accuracy with 4.2x PTE growth (severe over-thinking)

This indicates thinking mode is beneficial only when task difficulty justifies the extra compute.

## Limitations & Future Work

PTE measures transformer computation, omitting real-world costs like API latency. The γ parameter represents a simplified abstraction; while useful, it may not capture all nuances of specific hardware optimizations. Validation was limited to specific tasks and models; generalizability of the low PTE-high quality correlation requires further exploration.
